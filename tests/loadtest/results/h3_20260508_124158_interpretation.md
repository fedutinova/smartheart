## Interpretation

The held-out run supports the performance part of H3: the hybrid cache returns answers for 33 of 50 unseen formulations, so `Cache_hit_rate_heldout = 0.66`, above the 0.30 threshold.

The latency difference is also visible: cached responses are sub-second on this stand (`P95_cached = 985 ms`), while uncached responses remain multi-second (`P95_uncached = 9065 ms`). Because the run uses the real RAG/LLM path rather than fixed `RAG_MOCK_DELAY`, absolute values should be interpreted as stand-specific, not production-general.

Note on the negative run summary: avg duration (1200 ms) exceeds P95 (1003 ms), which is counterintuitive. This is explained by the bimodal distribution — 23 of 25 requests are fast cache hits (~689 ms each), and only 2 slow misses (~6858 ms) pull the mean above P95. P95 corresponds to the 24th fastest request, which is still a cache hit.

However, the negative/adversarial run exposes a serious safety issue: 23 of 25 adversarial questions were served from cache. This means the current vector threshold (`0.88`) is too permissive for medical cache reuse. The system is useful as an acceleration mechanism, but not yet safe enough to claim robust semantic equivalence.

### Fill run anomaly

6 of 25 requests during the fill phase were served as HITs. The cache should have been empty at that point. Either the cache was not cleared before the run (residual data from prior experiments), or the fill dataset contains internal near-duplicates semantically close enough to match each other. This needs to be controlled: add an explicit `TRUNCATE kb_cache` step before the fill phase in the test protocol.

## Methodological conclusion

H3 should be interpreted in two parts:

1. **Effectiveness subclaim — supported.** Hit-rate on held-out formulations is 0.66, above the 0.30 threshold. Cached responses are ~10× faster than full RAG (P95: 985 ms vs 9065 ms).
2. **Safety subclaim — not supported.** Adversarial false-hit rate is 0.92. The threshold-based approach cannot fix this.

### Why threshold tuning alone does not work

The adversarial questions are *genuinely* high-similarity to their cached counterparts — the embedding model correctly identifies "left vs right ventricle" or "with vs without ST elevation" as semantically close, because they share nearly all tokens. Raising the threshold eliminates true-positive hits faster than false positives:

| Threshold | Measure hit rate | Neg false-hit rate |
|----------:|:----------------:|:-----------------:|
| 0.88 | **0.66** ✓ | 0.92 ✗ |
| 0.93 | 0.18 ✗ | 0.36 ✗ |
| 0.95 | 0.06 ✗ | 0.32 ✗ |
| 0.97 | 0.00 ✗ | 0.24 ✗ |

There is no threshold at which both H3 requirements are simultaneously satisfied.

### Why AND(vector, trigram) does not work either

Antonym pairs also have high trigram similarity: "левого желудочка" vs "правого желудочка" scores 0.83, "с подъёмом сегмента ST" vs "без подъёма" scores 0.84. Requiring AND(vector ≥ 0.88, trigram ≥ 0.40) drops the measure hit rate to 0.12 while leaving 13 of 25 negative HITs in place.

### What the evidence points to

The only mechanism capable of distinguishing antonym pairs while preserving reformulation hits is **domain-specific contradiction detection**: a guard that checks whether the incoming question and the cached question differ on a known medical antonym axis (left/right, long/short QT, hyper/hypo, with/without ST elevation, first/second/complete block). This is not a threshold; it is a pre-filter that vetoes a match regardless of similarity score.

The honest conclusion is: the hybrid `pg_trgm + pgvector` cache is technically effective as an acceleration mechanism. The safety subclaim requires a contradiction guard layer that is not yet implemented.

## Recommended next iteration

1. **Implement contradiction guards** (pre-filter, not threshold). Define a list of antonym pairs for the cardiac-ECG domain. Before accepting a cache hit, check whether the incoming question and cached question differ on any antonym axis. A single mismatch on a guarded pair → force MISS regardless of similarity. This directly addresses the antonym category (highest-risk false hits) without degrading the hit rate on reformulations.
2. **Rerun the negative suite against the guarded implementation** to measure residual false-hit rate. Target: < 0.10.
3. **Clean up the test protocol**: add `TRUNCATE kb_cache` before the fill phase to eliminate ambiguity in the fill-run hit rate.
4. **Activate the pg_trgm path**: the current implementation never triggers trigram-based matching. Either wire it up or remove the dead branch; the "hybrid" label is currently misleading.
5. **Store false-hit pairs from `negative` as a permanent calibration set.** Any future threshold or guard change must be validated against this set before reporting results.
